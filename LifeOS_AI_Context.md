# LifeOS 系统架构背景

此文档整理了 LifeOS 系统的整体架构、文件结构、核心原则和控制机制，作为 AI 可以理解和参考的背景知识。内容基于当前文件系统和对话历史。

## 1. 系统概述
LifeOS 是一个个人生活操作系统，旨在通过结构化的文件夹和文件管理健康、关系、工作等领域。架构分为多个层级，包括治理层 (1_ControlPlane)、领域层 (3_Domains)、探索层 (4_Exploration)、信息流层 (0_Inbox/2_DAO/7_Archive)、工具层 (5_Tools) 和资源层 (6_Resources)。

关键概念：
- **三件套模板**：Service (服务目标)、Control (决策规则)、DAO (数据支持)，用于标准化领域实现。
- **注意力时间线**：DAO 作为注意力分配的时间线记录载体，支持系统决策。
- **SERVICE-E 元原则**：能量第一性，作为系统存在论基础，优先级最高。

## 3. Service Constitution (宪法)
位于 [📜_Service_Constitution.md](file:///d:/个人记录/obsidian-file/wjmber/tasks/需求/LifeOS/1_ControlPlane/📜_Service_Constitution.md)。

### 0. 【SERVICE-E 元原则】（系统存在论基础）
1. **能量第一性**： 我的意识质量、决策能力和“自我”的稳定性，直接受制于我的细胞能量（ATP）供应水平。我不是“思考的我”，我是“能量允许我去思考的我”。
2. **CEO 职责**： 我的首要职责是担任自身“能量发电站”的 CEO，负责优化能量的输入（饮食、睡眠、呼吸、环境）。
3. **调度官职责**： 我的第二职责是担任“认知资源”的调度官，负责规划能量的输出（注意力分配、任务管理、情绪控制）。

**此原则的优先级高于所有其他 SERVICE 原则。任何与之冲突的规则都必须重新审视。**

### 一、最高级原则（不可讨论）
1️⃣ **稳定优先于表现**  
任何短期表现提升，若以破坏能量稳定、情绪稳定、节律稳定为代价，长期必然亏损。  
→ **含义**：不为“今天更猛”牺牲“未来可用”。

2️⃣ **注意力是第一稀缺资源**  
金钱、信息、机会，只有在注意力稳定的前提下才有意义。  
→ **含义**：凡是高刺激、高波动、高比较的事物，默认视为成本，而非收益。

3️⃣ **系统完整性高于单次最优**  
允许单次判断错误，但不允许为了当下方便破坏系统结构。  
→ **含义**：不为省事篡改事实，不为冲动重写规则。

### 二、健康与能量原则（Health Domain）
4️⃣ **血糖稳定是能量稳定的基础**  
避免频繁、剧烈的血糖波动，以维持认知清晰与情绪可控。  
→ **接受的代价**：放弃部分即时口感、便利饮食与社交随意性。

5️⃣ **呼吸、睡眠、光照优先于补剂**  
所有外源补充（咖啡因、MCT、营养品）只能作为工具，而非基础。  
→ **含义**：不靠“hack”长期硬撑系统。

6️⃣ **运动的目标是代谢能力，而非自我证明**  
运动服务于线粒体、心肺和恢复力，而不是形象或自律表演。  
→ **含义**：不过度追求强度，不用疲惫换成就感。

### 三、工作与认知原则（Work & Cognition）
7️⃣ **深度产出优于持续忙碌**  
长时间低质量消耗，被视为系统故障，而非勤奋。  
→ **含义**：允许空白、暂停、节律性休息。

8️⃣ **复杂性是最后的手段**  
在问题被证明无法用简单结构解决前，不引入新系统、新工具、新概念。  
→ **含义**：先减法，后加法。

### 四、探索与不确定性原则（Exploration）
9️⃣ **探索必须在“沙盒”中进行**  
任何未经长期验证的想法、策略、投资，默认放入实验区，而非核心系统。  
→ **含义**：允许失败，但不污染主系统。

🔟 **投资的目的不是赢钱，而是保持对结构的理解**  
若某种参与方式主要消耗注意力与情绪，即使赚钱也视为失败。  
→ **含义**：我不靠刺激维持与世界的连接。

### 五、宪法修改规则（非常重要）
1️⃣1️⃣ **Service 可以被修改，但必须满足三个条件**  
不在强刺激当下、有 DAO 事实支持、修改记录被保留，不覆盖旧版本。  
→ **含义**：宪法可以进化，但不能被当场篡改。

### 【一句话版 Service（给 Control 调用）】
**我的长期目标不是最大化短期表现，而是维持一个低波动、可持续、可修正的人生系统。**

## 4. Domain Control 机制
Control 机制统一采用四个字段：触发条件、适用范围、判断规则、输出。以下是三个核心领域的 Control 规则。

### Health Domain ([🚦_Control.md](file:///d:/个人记录/obsidian-file/wjmber/tasks/需求/LifeOS/3_Domains/Health/🚦_Control.md))
| 字段 | 内容描述 |
| :--- | :--- |
| **触发条件** | 每日 07:00 唤醒检测 OR 任意时刻检测到心率变异性(HRV)异常 |
| **适用范围** | 物理身体的所有主动消耗行为（HIIT、力量训练、深度脑力工作） |
| **判断规则** | 1. **睡眠红线**：昨晚睡眠不足 6 小时，则**丢弃**今日所有高强度训练。<br>2. **生理预警**：检测到 HRV 异常或极度疲劳，则所有消耗项**延迟**至下一检测时段。<br>3. **执行标准**：状态评分 > 70% 且处于计划日，则**执行**既定项目。<br>4. **默认处理**：其余模糊情况一律**丢弃**。 |
| **输出** | **执行 / 延迟 / 丢弃** |

### Relationships Domain ([🚦_Control.md](file:///d:/个人记录/obsidian-file/wjmber/tasks/需求/LifeOS/3_Domains/Relationships/🚦_Control.md))
| 字段 | 内容描述 |
| :--- | :--- |
| **触发条件** | 收到社交邀请 OR 计划内关系维护时段 |
| **适用范围** | 所有涉及他人的互动（聚会、聊天、合作） |
| **判断规则** | 1. **成本红线**：预计情绪/能量消耗 > 当前储备 20%，则**丢弃**。<br>2. **价值评估**：若互动能带来长期关系资本，则**执行**。<br>3. **模糊区**：无明确收益或成本过高，则**延迟**至低峰期。<br>4. **默认**：优先保护个人节奏。 |
| **输出** | **执行 / 延迟 / 丢弃** |

### Work Domain ([🚦_Control.md](file:///d:/个人记录/obsidian-file/wjmber/tasks/需求/LifeOS/3_Domains/Work/🚦_Control.md))
| 字段 | 内容描述 |
| :--- | :--- |
| **触发条件** | 新任务进入 OR 每日 09:00 工作规划 |
| **适用范围** | 所有职业相关活动（会议、编码、学习） |
| **判断规则** | 1. **ROI 红线**：预计回报 < 投入成本 1.5 倍，则**丢弃**。<br>2. **专注匹配**：任务匹配当前认知状态，则**执行**。<br>3. **过载保护**：任务堆积 > 3，则低优先级**延迟**。<br>4. **默认**：优先高杠杆任务。 |
| **输出** | **执行 / 延迟 / 丢弃** |
